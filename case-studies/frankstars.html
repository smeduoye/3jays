The problem

A busy solo barber receives booking requests across WhatsApp, SMS, phone, and walk-ins. Messages arrive vague, overlapping, and easy to lose. Family bookings, poor timekeepers, and off-peak flexibility all add mental load — time spent parsing text instead of cutting hair or planning the day.

Generic booking apps ask customers to change how they message. That friction does not fit a neighbourhood chair where trust is personal and WhatsApp is already the channel.

The proposal

A private booking intelligence layer: messages in, structured requests in a single inbox, slot suggestions the barber can approve, edit, or reject, and clear proposals back to customers. Fixed slots for peak times; floating windows for off-peak flexibility. Every allocation requires a two-party handshake — the barber's proposal, then the customer's explicit acceptance.

What you'll review in the deck

The interactive proposal walks through the full experience — twelve slides covering the problem, design principles, booking state machine, inbox workflow, fixed and floating bookings, calendar view, counter-offers, and the customer experience. No app download for clients; the barber keeps authority at every step.

  • Human in the loop — no automatic bookings; every outbound proposal passes the barber first
  • Two-party confirmation — allocated only when the customer accepts a proposal the barber sent
  • Language parsing, deterministic calendar — AI interprets messy messages; slot logic stays auditable
  • Fixed and floating slots — pinned times when precision matters; windows for off-peak utilisation
  • Counter-offers return to inbox — negotiation stays with the barber; the system tracks state

MVP scope

The first release focuses on proving value without over-automation:

  • Message ingest from WhatsApp, SMS, and quick manual capture
  • Parsed inbox cards with confidence, ambiguities, and raw message always visible
  • Slot suggestions — fixed and floating — with barber approve / edit / reject
  • Outbound proposals and inbound acceptance handling
  • Calendar with fixed blocks and floating window bands
  • Customer notes, service durations, working hours, and proposal expiry
